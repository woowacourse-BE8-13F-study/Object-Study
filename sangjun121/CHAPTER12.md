## 12장

- 상속을 단순히 코드를 재사용하기 위한 목적으로 사용하면 안된다. 상속은 타입 계층을 구조화하기 위한 용도로 사용해야 한다.(389p)
- 이번 12장은 다형성이 어떻게 구현되는지 기술적인 매커니즘을 디테일하게 살펴본다.
    - 다형성은 런타임에 메시지를 처리하기에 적합한 메서드를 동적으로 탐색하는 과정을 통해 구현된다. 상속은 이런 메서드를 찾기 위한 일종의 탐색 경로를 클래스 계층의 형태로 구현하기 위한 방법이다.

### 다형성의 종류

- 다형성: 여러 타입을 대상으로 동작할 수 있는 코드를 작성할 수 있는 방법(390p)
- 유니버설 다형성: 타입이 달라도 같은 방식으로 처리하는 것
    - 매개변수 다형성
        - 일반적으로 제네릭을 떠올리면 편하다. 매개변수 타입을 사용 시점에 구체적인 타입으로 지정하는 방식을 의미한다.
    - 포함 다형성(서브타입 다형성)
        - 수신한 객체의 타입에 따라 실제로 수행되는 행동이 달라지는 능력을 의미한다.
        - 우리가 객체지향에서 일반적으로 말하는 다형성을 의미한다.
- 임시 다형성: 타입에 따라 다른 구현을 선택하는 것
    - 오버로딩 다형성:
        - 하나의 클래스 안에 동일한 이름의 메서드가 존재하는 것
    - 강제 다형성:
        - 언어가 지원하는 자동적인 타입 변환이나, 사용자가 직접구현한 타입변환을 이용해 동일한 연산자를 다양한 타입에 사용하는 방식
        - 예를 들어, + 연산자의 앞뒤에 문자형과 정수형이 오는 경우, 정수형이 자동으로 문자형으로 캐스팅된다.

### 상속을 바라보는 관점

- 포함 다형성을 위한 전제조건은 자식 클래스가 부모클래스의 서브 타입이어야 한다. 결국 상속의 진정한 목적은 **다형성을 위한 서브타입 계층을 구축하기 위함이다.**
- 포함 다형성을 위해 상속을 사용하는 가장 큰 이유는, 상속이 클래스들을 계층으로 쌓아올린 후 상황에 따라 적절한 메서드를 선택할 수 있는 매커니즘을 제공하기 때문이다. 결국 이 계층 구조 덕분에 적절한 메서드를 탐색할 수 있다.
- 이번 장에서는 포함 다형성을 구현하는 방법을 설명하기 위해 상속을 예시로 설명한다. 즉, 이번 장에서 다루는 매커니즘은 상속 외에도 포함 다형성을 구현할 수 있는 다양한 방법에 공통적으로 적용할 수 있는 개념이라는 사실을 기억해야 한다.

### 상속 매커니즘을 이해하는데 필요한 5가지 개념

- 업캐스팅, 동적 메서드 탐색, 동적 바인딩, self 참조, super 참조에 대해 살펴보자.

### 자식과 부모 사이의 상속 개념 2가지

1. 데이터의 상속: 상속을 사용한다면, 자식 클래스의 인스턴스는 자동으로 부모 클래스에서 정의한 인스턴스 변수를 내부에 포함하게 된다.
2. 행위의 상속: 런타임에서 시스템이 자식 클래스에 정의되지 않은 메서드가 있을 경우, 이를 부모 클래스에서 탐색한다.

여기서 행위의 상속에서 시스템이 메서드를 찾는 과정에 대해 깊이 있게 알아보자.

### 상속 관계에서 메서드 탐색 과정

- 우선, 상태와는 다르게, 메서드의 경우에는 동일한 클래스의 인스턴스끼리 공유가 가능하기 때문에, **클래스는 한번만 메모리에 로드하고 각 인스턴스별로 클래스를 가리키는 포인터를 갖는 것이 경제적이다.**
- 즉, 각 인스턴스는 class라는 포인터로 자신의 클래스를 가르키고 있으며, 해당 클래스는 자신의 부모 클래스를 parent이라는 포인터로 참조하고 있다.
- 인스턴스에 메세지 요청이 들어오면(메서드 호출) 해당 포인터를 타고 메서드의 탐색 과정을 수행한다.

### 업캐스팅과 동적 바인딩

- 코드 안에서 선언된 참조 타입과 무관하게 실제로 메시지를 수신하는 객체의 타입에 따라 실행되는 메서드가 달라질 수 있는 이유는, 업캐스팅과 동적 바인딩이라는 매커니즘이 작동하기 때문이다.(405p)
- 업캐스팅: 부모 타입으로 선언된 변수에 자식 클래스 인스턴스를 할당할 수 있는 것
- 동적 바인딩(지연 바인딩): 시스템이 메시지를 처리할 메서드를 컴파일 시점이 아닌, 실행 시점에 결정하는 것
- 개방 폐쇄 원칙이 목적이라면, 업캐스팅과 동적바인딩은 수단이다.
- 참고. 다운 캐스팅이란, 부모 클래스의 인스턴스를 자식 클래스 타입으로 변환하기 위해 명시적으로 캐스팅하는 것
- **즉, 업캐스팅과 동적 바인딩을 사용하면, 부모 클래스 참조에 대한 메시지 전송을 자식 클래스에 대한 메서드 호출로 변환할 수 있다.**

### 동적 메서드 탐색과 다형성

- 메시지를 수신한 객체는 먼저 자신을 생성한 클래스에 적합한 메서드가 있는지 검사한다. 없다면, 부모 클래스에서 탐색을 이어간다. 최상위 클래스에서도 발견하지 못한다면 예외를 발생시키거나 후작업을 통해 탐색을 중단한다.
- 메서드 탐색의 핵심 변수는 self 참조이다. 즉, 컴파일러는 임시변수인 self를 자동으로 생성하고 메시지를 수신한 객체를 가르키도록 설정한다. 메서드 탐색이 종료되는 순간 해당 self는 자동 소멸된다.
- 즉, 메서드 탐색의 시발점은 self에 저장된 인스턴스의 주소이다. 이후 class 포인터와 parent포인터를 타고 메서드를 탐색한다.
- 메서드의 탐색 방향은 자식에서 부모클래스이다. 이에 따라 자식 클래스에 선언된 메서드가 부모클래스보다 우선순위가 높다.
- 동적 메서드 탐색은 2가지 원리로 구성된다. (409p)
    1. 자동적인 메시지 위임: 자신이 이해할 수 없는 메시지를 받은 경우 상속 계층을 따라 부모클래스에게 처리를 위임한다.
    2. 메서드 탐색을 위해 동적인 문맥을 사용한다: 어떤 메서드가 실행될지는 컴파일 시점이 아닌, 실행시점에 이뤄지며 메서드를 탐색하는 경로는 self참조를 이용해서 결정한다.
    - 즉, 런타임에 실제로 메시지를 수신할 객체가 어떤 타입인지 찾아야하며, 해당 타입에 따라 메서드를 탐색하는 첫 시발점이 런타임에(동적으로) 결정된다. 여기서 가장 중요한 역할을 하는 것이 self이다.

 

### 메서드 오버로딩과 오버라이딩

- 이 두가지 매커니즘을 동적 메서드 탐색의 관점에서 바라보자.
- 메서드 오버라이딩은 모든 객체 지향언어에서 유사하게 동작하는데 비해, 메서드 오버로딩은 언어에 따라 조금씩 달라진다.(410p)
- 오버라이딩은 단순하다. 자식 클래스에서 오버라이딩된 메서드가 있다면, 메서드 탐색 순서가 자식 클래스가 우선이기 때문에 해당 메서드를 실행하고 탐색을 종료하게 된다. 즉, 이를 책 표현을 빌리자면, “자식 클래스에서 부모 클래스로 향하는 메서드 탐색 순서 때문에 자식 클래스가 부모 클래스의 메서드를 감추게 된다”(413p)라고 표현할 수 있다.
- 반면 메서드 오버로딩은 자식클래스가 부모 클래스의 메서드를 감추지는 않는다. 메서드 오버라이딩은 메서드를 감추지만 메서드 오버로딩은 사이좋게 공존한다.(414p)
- C++에서는 상속 계층 사이의 메서드 오버로딩을 지원하지 않는다. 동일 클래스 내부에서만 오버로딩을 지원한다. 즉, 부모 클래스에 선언된 이름이 동일한 메서드 전체를 숨겨서 클라이언트가 호출할 수 없게 만들고 이를 이름 숨기기라고 한다. 이름 숨기기 문제를 해결하는 방법은 부모 클래스에 정의된 모든 메서드를 자식 클래스에 오버로딩하는 것이다.

### Self 전송

- self는 동적으로 특정 객체가 초기에 지정된다고 말했다. 즉, 현재 클래스의 메서드를 호출하는 것이 아니라, 현재 객체(self)에게 메세지를 전송하는 것이다.(417p)
- self 전송은 처음에 지정된 메시지를 전송받은 객체로 다시 돌아가는 것을 의미한다. 즉 메서드 탐색의 지점이 다시 self 영역으로 내려갔음을 의미한다.

### 처리할 수 있는 메서드가 없는 경우?

- 최상위 부모 클래스까지 탐색을 진행했는데, 메서드가 없는 경우는 어떻게 처리할 수 있을까? **이 처리 방법은 프로그래밍 언어가 정적 타입언어인지, 동적 타입언어인지에 따라 달라진다.**
- 정적 타입 언어: 코드를 컴파일할 때 판단한다. 즉 이해할 수 없는 메시지를 발견했을 때 컴파일 단계에서 에러가 발생한다. (자바는 정적 타입 언어이다.)
- 동적 타입 언어: 동적 타입 언어에는 컴파일 단계가 존재하지 않기 때문에 런타임 도중 예외가 발생한다. 즉, 실제로 코드를 실행해보기 이전에는 메시지 처리 가능 여부를 판단할 수 없다.
- 동적 타입 언어에서는 이해할 수 없는 메시지에 예외를 던지는 것 외에도, doesNotUnderstand 메시지에 응답할 수 있는 메서드를 구현할 수 있다.
- 정적 타입과 동적 타입 언어의 트레이드 오프를 알아보자. 동적 타입 언어의 동적인 특징은 유연성을 높여주지만 코드를 이해하고 수정하기 어렵게 만든다 또한 디버깅 과정을 복잡하게 만든다. 정적 타입 언어에는 이런 유연성이 부족하지만 안정적이다.

### Super 키워드

- 자식 클래스에서 부모 클래스의 메서드 구현을 사용해야 하는 경우에 super를 사용한다.
- 이때 부모 클래스에 해당 메서드가 존재하지 않을 수 있다. 부모클래스의 상위 클래스들 사이에 존재할 수 있다. 즉, super 참조의 정확한 의미는, “지금 이 클래스의 부모클래스부터 메서드 탐색을 시작하세요”이다. 이것은 super 참조를 통해 실행하고자 하는 메서드가 반드시 부모클래스에 위치하지 않아도 된다는 유연성을 제공한다.(424p)
- 다만, super 키워드의 경우 self키워드와 다르게 미리 정해진다. self키워드는 실행시점에 동적으로 정해지지만 super의 경우 컴파일 시점에 미리 결정해 둘 수 있다. (언어에 따라 super가 런타임에 지정되는 경우도 있다.)

### 위임과 포워딩의 차이점

- 위임은 본질적으로 자신이 정의하지 않거나, 처리할 수 없는 속성 또는 메서드의 탐색 과정을 다른 객체로 이동시킬 때 사용된다. 이를 위해 위임은 항상 현재 실행 문맥을 가르키는 self 참조를 인자로 전달한다. 이때 self를 넘기는 이유는 클래스를 이용한 상속간계를 객체사이의 합성 관계로 대체해서 다형성을 구현하기 위함이다.
- 포워딩은 self 참조를 전달하지 않는 것을 의미한다.
- 간단하게 GradeLecture을 Lecture의 자식 클래스로 선언하면, 실행시에 인스턴스 사이에서 self 참조가 자동으로 전달된다. 이 참조의 전달이 결과적으로 자식 클래스의 인스턴스와 부모 클래스의 인스턴스 사이에 동일한 실행 문맥을 공유하게 만들어 준다.
- 즉, 상속은 동적으로 메서드를 탐색하기 위해 현재 실행 문맥을 가지고 있는 self 참조를 전달한다.

### 프로토타입 기반 객체 지향 언어

- 클래스가 아닌 객체를 이용해서도 상속을 흉내낼 수 있다. 오직 객체만 존재하는 프로토타입 기반의 객체지향 언어에서는 상속을 규현하는 유일한 방법이 객체 사이의 위임을 이용하는 것이다.(430p)
- 자바스크립트가 대표적인 예시이며, 클래스가 존재하지 않기 때문에 오직 객체들 사이의 메시지 위임만을 이용해 다형성을 구현한다. 이는 객체지향 패러다임에서 클래스가 필수 요소가 아님을 보여준다. 상속 이외의 방법으로도 다형성을 구현할 수 있다는 사실도 보여준다.
- 정리하자면, 프로토타입 기반의 객체지향 언어는 객체 사이의 자동적인 메시지 위임을 통해 상속을 구현한다. 중요한 점은 클래스 기반 상속과 객체 기반의 위임 사이에 기본 개념과 매커니즘을 공유한다는 점이다.
